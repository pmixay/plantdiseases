## Блок 1 — Обученная модель (КРИТИЧНО)

Без обученной модели проект — демо-заглушка. Это **единственный блокер** между текущим состоянием и рабочим приложением.

### 1.1 Обучение моделей

- [ ] Скачать PlantVillage dataset (Kaggle или GitHub)
- [ ] Запустить Colab-ноутбук `train_notebook.ipynb` на GPU (T4)
- [ ] Убедиться что маппинг папок PlantVillage → 15 классов корректен (ячейка 3 ноутбука)
- [ ] Обучить Detector (Stage 1) — target: ≥ 92% binary accuracy
- [ ] Обучить Classifier (Stage 2) — target: ≥ 80% на 15 классах
- [ ] Скачать `detector.pth` + `classifier.pth` + `classes.json`
- [ ] Положить в `server/models/`, перезапустить сервер
- [ ] Проверить `/api/health` — `pipeline_mode: "full"`

### 1.2 Валидация модели

- [ ] Проанализировать confusion matrix — найти классы с accuracy < 70%
- [ ] Проверить на реальных фото комнатных растений (не из датасета) — минимум 20 фото
- [ ] Замерить latency: target < 2 сек на CPU для полного пайплайна
- [ ] Если accuracy < 80% — добавить фото из iNaturalist для слабых классов и переобучить

### 1.3 Фикс известных багов

- [ ] Исправить `server/model.py` — убрать сломанный импорт `CLASS_NAMES` (его нет в classifier.py на уровне модуля)
- [ ] Синхронизировать LICENSE и README (выбрать одну лицензию)
- [ ] Удалить или реализовать Docker (не отмечать как ✅ то, чего нет)
- [ ] Обновить ROADMAP.md — заменить "MobileNetV2" на "MobileNetV3-Small"

## Блок 2 — Полировка UX до production-grade

Разница между студенческим проектом и production — в деталях. Эти пункты делают приложение user-friendly и впечатляют жюри.

### 2.1 Onboarding и первое впечатление

- [ ] Splash screen (Android 12+ SplashScreen API) — иконка растения, зелёный фон, < 1 сек
- [ ] Onboarding-экран при первом запуске (3 слайда ViewPager2): «Сканируй → Узнай болезнь → Получи лечение». ViewPager2 уже в зависимостях
- [ ] Анимация Lottie на экране анализа (зависимость уже подключена, но не используется) — заменить CircularProgressIndicator на анимацию сканирования растения

### 2.2 Экран результата — wow-эффект

- [ ] **Grad-CAM визуализация на Android** — показать тепловую карту поверх фото (сервер уже возвращает `detection.region`, нужно нарисовать overlay). Это главная фишка для жюри: «смотрите, модель показывает ГДЕ именно болезнь»
- [ ] Анимированное появление результата (раскрытие карточек снизу вверх)
- [ ] Иконки для каждого шага лечения и профилактики (вместо голых цифр)
- [ ] Кнопка «Новый скан» на экране результата (сейчас нужно нажимать Back)

### 2.3 Улучшения камеры

- [ ] Фильтр «не растение» — если на фото нет зелёного (HSV анализ на клиенте), показать предупреждение «Это не похоже на растение» перед отправкой на сервер
- [ ] Автофокус при тапе на preview (CameraX поддерживает)

### 2.4 Галерея

- [ ] Пустое состояние с анимацией (Lottie), а не просто текст
- [ ] Карточки — добавить дату в более читаемом формате («Сегодня», «Вчера», «3 дня назад»)

### 2.5 Справочник

- [ ] Поиск по справочнику (SearchView в toolbar или поле поиска сверху)
- [ ] Иконки — заменить одинаковые `ic_disease` / `ic_pest` на уникальные для каждой категории (можно простые цветные кружки с буквами)

### 2.6 Тёмная тема

## Блок 3 — Надёжность и Production-grade

### 3.1 Обработка ошибок и сеть

- [ ] Retry с экспоненциальным backoff при сетевых ошибках (OkHttp Interceptor)
- [ ] Timeout-индикация — если сервер не ответил за 10 сек, показать «Сервер не отвечает»
- [ ] Health-check при запуске — проверить `/api/health` и показать статус сервера в профиле (зелёный/красный индикатор)
- [ ] Offline-banner — если нет сети, показать сообщение на экране сканера

### 3.2 Оптимизация производительности

- [ ] **Кэширование изображений** — Glide уже подключен, но добавить DiskCacheStrategy.ALL для галереи
- [ ] **Сжатие при загрузке** — `ImageUtils.prepareImageForUpload` сжимает до 1024px — проверить что EXIF-ориентация не теряется на всех устройствах
- [ ] **Room индексы** — добавить индекс на `timestamp` и `is_healthy` для быстрой фильтрации
- [ ] **ProGuard** — добавить правила для Lottie, проверить release-сборку

### 3.3 Серверная надёжность

- [ ] Логирование запросов в файл (RotatingFileHandler) — для демонстрации жюри
- [ ] Rate limiting — защита от спама (1 запрос/сек на IP)
- [ ] CORS — сузить `allow_origins` с `*` до конкретных (на продакшене)
- [ ] Health endpoint — добавить uptime и количество обработанных запросов

### 3.4 Docker

- [ ] Написать `Dockerfile` (Python 3.11-slim + CPU PyTorch + requirements)
- [ ] Написать `docker-compose.yml` (один сервис, маппинг порта 8000, volume для моделей)
- [ ] Проверить что сервер стартует в контейнере с обученными моделями

Это то, что выделит проект среди других. Каждый пункт — потенциальный «вау-момент» на защите.

Сервер уже генерирует bounding box через Grad-CAM. Нужно:
- [ ] На Android: нарисовать полупрозрачный цветной прямоугольник на фото в экране результата (`detection.region` → `Canvas.drawRect`)

- [ ] Для защиты: подготовить скриншот «до и после» — фото → тепловая карта → диагноз

### 4.2 Метрики и аналитика для жюри

- [ ] Экран «О модели» в профиле: архитектура (MobileNetV3 + EfficientNet), количество параметров (8M), количество классов, accuracy на валидации
- [ ] При демонстрации: показать confusion matrix из ноутбука
- [ ] Подготовить 5-7 тестовых фото с разными болезнями для live-demo

### 4.3 Объяснимость результатов

- [ ] На экране результата: «Почему мы так решили?» — блок с топ-3 альтернативными диагнозами и их вероятностями (сервер уже возвращает `all_probs` — нужно передать на Android)
- [ ] Показать confidence bar для каждого альтернативного диагноза

### 4.4 Двухъязычный live-demo

- [ ] Подготовить сценарий демонстрации: переключение языка в 1 тап
- [ ] Убедиться что ВСЕ экраны корректно отображаются на обоих языках (особенно результат и справочник)
